### GUARD-001: Non-GuardrailViolationError Exceptions Crash Pipeline

**Severity**: HIGH
**Phase**: 4
**Found by**: reviewer2
**Status**: FIXED

**Description**:
The `GuardrailMiddleware` in `middleware.py` only catches `GuardrailViolationError` exceptions in `run_pre_call()` and `run_post_call_success()`. If a guardrail raises any other exception (e.g., `AttributeError`, `KeyError`, `ConnectionError` for Lakera API), the exception propagates and crashes the entire request pipeline.

**Files Impacted**:
- `src/guardrails/middleware.py` (lines 30-43, 59-69)

**Suggested Fix**:
Wrap all guardrail hook calls in a broad exception handler that treats unexpected errors as either fail-open or fail-closed based on configuration:

```python
except GuardrailViolationError:
    if guardrail.action == GuardrailAction.LOG:
        logger.warning(...)
        continue
    raise
except Exception as exc:
    logger.exception("Guardrail failed unexpectedly", extra={"guardrail": guardrail.name})
    if guardrail.action == GuardrailAction.BLOCK:  # fail-closed
        raise GuardrailViolationError(...)
    # fail-open: continue
```

**Actual Fix**:
- Added broad exception handling in `src/guardrails/middleware.py` for both `run_pre_call()` and `run_post_call_success()`.
- For guardrails with `action=LOG`, unexpected exceptions now log and continue (fail-open).
- For guardrails with `action=BLOCK`, unexpected exceptions are mapped to `GuardrailViolationError` with `code=guardrail_execution_error` (fail-closed) instead of crashing the pipeline with raw exceptions.
- Added regression tests in `tests/guardrails/test_framework.py` covering unexpected pre-call and post-call exceptions.

### RATE-001: TPM Rate Limit Bypass via Malformed Request Body

**Severity**: HIGH
**Phase**: 6
**Found by**: reviewer2
**Status**: FIXED

**Description**:
In `rate_limit.py`, when `request.body()` raises a `RuntimeError` (lines 42-45), the `body` is set to empty bytes and `tokens` becomes 0. This allows an attacker to send malformed requests that bypass TPM (tokens per minute) rate limiting entirely, while still consuming actual tokens at the provider.

**Files Impacted**:
- `src/middleware/rate_limit.py` (lines 42-46)

**Suggested Fix**:
When body parsing fails, either reject the request or use a conservative token estimate (e.g., max body size):

```python
try:
    body = await request.body()
    request._body = body
except RuntimeError:
    # Cannot parse body - use conservative estimate or reject
    raise InvalidRequestError("Could not parse request body")
```

**Actual Fix**:
- Confirmed `request.body()` `RuntimeError` handling previously fell back to a tiny token estimate (empty body path), which can materially undercount TPM and weaken enforcement.
- Updated `src/middleware/rate_limit.py` to reject unreadable request bodies by raising `InvalidRequestError("Could not parse request body for rate limiting")` instead of estimating tokens from an empty body.
- Added regression test in `tests/test_rate_limit.py` (`test_rate_limit_rejects_unreadable_request_body`) to ensure unreadable bodies are rejected.
